# Meetily 二次开发说明

## 项目目标

基于 Meetily 二次开发一个中文企业会议转录和纪要助手，主要用于 Microsoft Teams 会议。

不要从零重写项目。保留 Meetily 原有底层能力，包括：
- 音频捕获
- 本地转录
- 实时转录
- AI Summary
- 自定义 Provider
- 本地模型支持
- Tauri 打包能力

主要改造方向：
- 中文化 UI
- macOS 风格三栏布局
- DeepSeek 快捷接入
- 自定义 API 配置增强
- 中文企业会议纪要模板
- 导出能力优化

## UI 目标

整体 UI 使用 macOS 风格三栏布局。

### 左侧栏

包含：
- 搜索会议记录
- 当前会议卡片
- 录音状态
- 开始录音按钮
- 停止按钮
- 历史会议列表

### 中间主区域

包含：
- 会议标题
- 会议日期
- 参与人数
- 录音时长
- 导入音频按钮
- 三个标签页：
  - AI 纪要
  - 原始转录
  - 待办事项

AI 纪要包括：
- 会议摘要
- 领导要求
- 风险点
- 关键决策

原始转录包括：
- 说话人
- 时间戳
- 发言内容

待办事项包括：
- 事项
- 负责人
- 截止日期
- 状态

### 右侧栏

包含：
- 转录模式选择
  - 本地 Whisper
  - 云端转录 API
- 自定义 API 配置
  - 转录服务
  - 转录 Base URL
  - 转录模型名
  - 纪要服务
  - 纪要 Base URL
  - 纪要模型名
  - API Key
  - 测试连接按钮
- 隐私与安全说明
  - API Key 保存到本地安全存储
  - 会议音频默认本地保存
  - 可选择只上传文字给 AI

## DeepSeek 支持

增加 DeepSeek 快捷 Provider。

默认配置：
- Provider: DeepSeek
- Base URL: https://api.deepseek.com/v1
- Model: deepseek-chat

要求：
- 底层使用 OpenAI-compatible Chat Completions 格式
- API Key 由用户填写
- 不要写死 API Key
- 不要在日志里打印 API Key
- 支持用户改成 deepseek-reasoner 或其他模型名
- 错误提示要清楚：
  - API Key 缺失
  - Base URL 错误
  - 模型不存在
  - 网络连接失败

## 中文会议纪要模板

增加模板：中文企业会议纪要。

输出格式：

# 会议主题
# 会议摘要
# 领导要求
# 关键决策
# 待办事项
| 事项 | 负责人 | 截止时间 | 状态 |
# 风险点
# 需要后续确认的问题

规则：
- 默认中文输出
- 不要编造负责人和截止时间
- 如果没有提到，写“未明确”
- 尽量提取明确的会议要求、整改项、风险点和后续跟进事项

## 开发规则

1. 不要大规模重构底层。
2. 不要删除原项目功能。
3. 不要删除已有测试，除非测试明显错误。
4. 修改后运行测试。
5. 如果没有相关测试，新增基础测试。
6. 不要引入不必要的新依赖。
7. 不要依赖外部 CDN 图标。
8. API Key 不允许硬编码。
9. API Key 不允许打印到日志。
10. 优先小步提交，每次只完成一个阶段。
11. 遇到构建错误，先分析原因，再修复。

## 第一阶段任务

不要直接全量重写。

先完成项目分析，输出：
1. 前端入口在哪里
2. 主要页面和组件在哪里
3. 设置页在哪里
4. AI Provider 逻辑在哪里
5. Summary Prompt 模板在哪里
6. 转录结果数据结构在哪里
7. DeepSeek Provider 应该加在哪里
8. 中文会议纪要模板应该加在哪里
9. 如果改成三栏 UI，最小改动方案是什么

## 第二阶段任务

在确认项目结构后，再开始做 UI 改造：
- 保留底层逻辑
- 重做主界面为 macOS 风格三栏布局
- 接入原有会议、转录、总结、设置数据
- 不要先做大规模功能重写

## 第三阶段任务

增加 DeepSeek 快捷 Provider 和自定义 API 配置。

## 第四阶段任务

增加中文企业会议纪要模板和相关测试。
